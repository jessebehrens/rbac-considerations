Snowflake RBAC Framework

A framework for thinking through Role-Based Access Control decisions in Snowflake.

This guide provides a starting point for designing your access control strategy. Adapt these patterns to fit your organization's unique security requirements and operational needs.

Core Principles

1. Reserve High-Privilege Roles

ACCOUNTADMIN and SECURITYADMIN should be used sparingly. Create custom admin roles (PLATFORM_ADMIN, FINOPS_ADMIN, DATA_SHARING_ADMIN, <LOB>_SYSADMIN, etc.) for daily operations. Follow your company's best practices for securing and auditing access to high-privilege accounts.

2. Separate Access from Function

Create Access Roles for object permissions. Create Functional Roles for job titles. Grant access roles to functional roles. Follow your company's naming conventions and best practices to ensure consistency and clarity across teams.

3. Plan Your Hierarchy Intentionally

Decide early whether custom roles should roll up to SYSADMIN. There's no one-size-fits-all answer—base this on your governance model, compliance needs, and operational structure. Start granular—it's easier to consolidate roles later than to split them apart.

Custom Admin Roles

These roles delegate specific administrative functions without granting full system role access.

PLATFORM_ADMIN

Manages warehouses, resource monitors, network policies, network rules, storage integrations, compute pools, session policies, password policies, authentication policies, notification integrations, tasks, and other Snowflake objects.

FINOPS_ADMIN

Monitors costs, manages budgets, and oversees resource consumption. Develops alerts for budget thresholds and cost anomalies. Has visibility into billing and usage metrics to optimize spend.

DATA_SHARING_ADMIN

Manages data shares, listings, and external data exchange. Controls where data is shared, but the database owner must separately grant objects to the share. This means DATA_SHARING_ADMIN can manage distribution without having query access to the data itself.

<LOB>_SYSADMIN

A line-of-business specific admin role that owns and manages databases, schemas, and objects for a particular business unit. Created outside the default role hierarchy so that custom roles and their access don't automatically roll up to ACCOUNTADMIN.
