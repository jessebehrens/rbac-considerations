RBAC Fundamentals

Understand the building blocks of Snowflake's access control model before designing your hierarchy.

The Access Control Triad

Snowflake's access control is built on three core concepts that work together:

Securable Objects

Anything you can grant access to: databases, schemas, tables, views, warehouses, stages, etc. Access is denied unless explicitly granted.

Roles

Containers for privileges. Roles are granted to users or to other roles. Users inherit all privileges from their granted roles.

Privileges

Specific permissions on objects: SELECT, INSERT, USAGE, OWNERSHIP, etc. In RBAC, privileges are always granted to roles, never directly to users.

How They Connect

The flow of access in Snowflake works like this:

-- 1. Privileges are granted on Objects to Roles
GRANT SELECT ON TABLE sales.orders TO ROLE analyst_role;

-- 2. Roles are granted to Users
GRANT ROLE analyst_role TO USER jane;

-- 3. Jane can now SELECT from sales.orders when using analyst_role

System-Defined Roles

Snowflake provides five built-in roles. Understanding their purpose prevents misuse:

Role Purpose Key Privileges When to Use
ACCOUNTADMIN Top-level account management All privileges + account settings Initial setup, billing, account parameters only
SECURITYADMIN Grant management MANAGE GRANTS globally Sparingly—use custom roles like PLATFORM_ADMIN instead
USERADMIN User and role creation CREATE USER, CREATE ROLE Creating users and roles (not granting privileges)
SYSADMIN Object management CREATE DATABASE, CREATE WAREHOUSE Creating and managing data objects
PUBLIC Default access for all users None by default Objects that everyone should access

System Role Hierarchy

The system roles have a built-in hierarchy:

ACCOUNTADMIN inherits privileges from both SECURITYADMIN and SYSADMIN.
SECURITYADMIN inherits from USERADMIN.

Critical Mistakes to Avoid
  • Never use ACCOUNTADMIN as any user's default role
  • Never create objects with ACCOUNTADMIN — they become hard to manage
  • Never use ACCOUNTADMIN in automated scripts
  • Avoid daily use of SECURITYADMIN — create custom admin roles without MANAGE GRANTS

The Principle of Least Privilege

Every RBAC design should start with this principle:

Grant only the minimum privileges necessary for users to perform their job functions.

Why This Matters

Reduces Blast Radius

If credentials are compromised, damage is limited to what that role can access.

Simplifies Auditing

Fewer grants mean clearer audit trails and easier compliance reviews.

Enforces Separation of Duties

Different roles for different responsibilities prevents conflicts of interest.

Supports Compliance

Required by GDPR, HIPAA, SOC 2, and other regulatory frameworks.

Object Hierarchy & Access Requirements

To access an object in Snowflake, you need privileges at every level of the hierarchy:

A common mistake is granting SELECT on a table but forgetting USAGE on the database and schema:

-- This alone won't work!
GRANT SELECT ON TABLE mydb.myschema.mytable TO ROLE reader;

-- You also need:
GRANT USAGE ON DATABASE mydb TO ROLE reader;
GRANT USAGE ON SCHEMA mydb.myschema TO ROLE reader;

-- And to run queries, warehouse access:
GRANT USAGE ON WAREHOUSE compute_wh TO ROLE reader;
Don't Forget Warehouses

Users need USAGE on a warehouse to execute queries, even if they have all the object privileges. This is a common source of "permission denied" errors.

Quick Reference: Which Role to Use

Task Role to Use
Create a new user USERADMIN
Create a new role USERADMIN
Grant role to user SECURITYADMIN
Grant role to another role SECURITYADMIN
Grant object privileges SECURITYADMIN or object owner
Create database or warehouse SYSADMIN
View billing / account settings ACCOUNTADMIN
