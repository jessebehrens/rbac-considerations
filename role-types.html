Access, Functional & Service Roles

The most important concept in RBAC design: organizing roles by their intent.

The Three-Tier Role Model

While Snowflake doesn't technically differentiate between role types, organizing roles by their intent creates a maintainable, scalable hierarchy. This is the most important concept in RBAC design.

Key Insight

Access roles define what can be accessed. Functional roles define who needs access. Users only receive functional roles—never access roles directly.

Access Roles (Building Blocks)

Purpose

Grant specific privileges on specific objects. These are the building blocks of your access model.

Naming Convention

{DATABASE}_{SCHEMA}_{LEVEL}_AR

Examples:

  • SALES_DB_READ_AR
  • SALES_DB_WRITE_AR
  • ANALYTICS_REPORTING_READ_AR

Access Role Rules

  • One access role per database/schema + permission level
  • Never granted directly to users
  • Contains only object privileges
  • Composed into functional roles

Standard Access Role Pattern

-- READ Access Role (SELECT only)
USE ROLE USERADMIN;
CREATE ROLE SALES_DB_READ_AR
    COMMENT = 'Read access to SALES_DB database';

USE ROLE SECURITYADMIN;
GRANT USAGE ON DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;
GRANT USAGE ON ALL SCHEMAS IN DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;
GRANT SELECT ON ALL TABLES IN DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;
GRANT SELECT ON ALL VIEWS IN DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;

-- Future grants for new objects
GRANT USAGE ON FUTURE SCHEMAS IN DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;
GRANT SELECT ON FUTURE TABLES IN DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;
GRANT SELECT ON FUTURE VIEWS IN DATABASE SALES_DB TO ROLE SALES_DB_READ_AR;
-- WRITE Access Role (DML operations)
USE ROLE USERADMIN;
CREATE ROLE SALES_DB_WRITE_AR
    COMMENT = 'Read-write access to SALES_DB database';

USE ROLE SECURITYADMIN;
GRANT USAGE ON DATABASE SALES_DB TO ROLE SALES_DB_WRITE_AR;
GRANT USAGE ON ALL SCHEMAS IN DATABASE SALES_DB TO ROLE SALES_DB_WRITE_AR;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN DATABASE SALES_DB 
    TO ROLE SALES_DB_WRITE_AR;

-- Future grants
GRANT USAGE ON FUTURE SCHEMAS IN DATABASE SALES_DB TO ROLE SALES_DB_WRITE_AR;
GRANT SELECT, INSERT, UPDATE, DELETE ON FUTURE TABLES IN DATABASE SALES_DB 
    TO ROLE SALES_DB_WRITE_AR;

Functional Roles (Business Alignment)

Purpose

Represent job functions within the organization. These are what you grant to end users.

Naming Convention

{TEAM}_{FUNCTION}_FR

Examples:

  • DATA_ENGINEER_FR
  • FINANCE_ANALYST_FR
  • MARKETING_MANAGER_FR

Functional Role Rules

  • Granted to end users
  • Composed of multiple access roles
  • Aligns with org chart/job titles
  • Should roll up to SYSADMIN

Creating Functional Roles

-- Create functional role for Data Analysts
USE ROLE USERADMIN;
CREATE ROLE DATA_ANALYST_FR
    COMMENT = 'Functional role for data analysts';

-- Grant access roles to the functional role
USE ROLE SECURITYADMIN;
GRANT ROLE SALES_DB_READ_AR TO ROLE DATA_ANALYST_FR;
GRANT ROLE MARKETING_DB_READ_AR TO ROLE DATA_ANALYST_FR;
GRANT ROLE ANALYTICS_DB_WRITE_AR TO ROLE DATA_ANALYST_FR;

-- Grant warehouse access
GRANT USAGE ON WAREHOUSE ANALYST_WH TO ROLE DATA_ANALYST_FR;

-- Roll up to SYSADMIN (important!)
GRANT ROLE DATA_ANALYST_FR TO ROLE SYSADMIN;

-- Grant to users
GRANT ROLE DATA_ANALYST_FR TO USER jane_smith;
GRANT ROLE DATA_ANALYST_FR TO USER bob_jones;

Service Roles (Non-Human Access)

Purpose

Grant access to external tools, ETL pipelines, and BI platforms. These are functional roles for machines.

Naming Convention

{TOOL/SERVICE}_SVC

Examples:

  • DBT_SVC
  • TABLEAU_SVC
  • FIVETRAN_SVC
  • AIRBYTE_SVC

Service Role Rules

  • Granted to service users (TYPE = SERVICE)
  • Composed of access roles (like functional)
  • Never granted to human users
  • Uses key pair auth, not passwords

Creating Service Roles

-- Create service role for Tableau
USE ROLE USERADMIN;
CREATE ROLE TABLEAU_SVC
    COMMENT = 'Service role for Tableau BI platform';

-- Grant only the access roles needed
USE ROLE SECURITYADMIN;
GRANT ROLE ANALYTICS_DB_READ_AR TO ROLE TABLEAU_SVC;
GRANT ROLE REPORTING_DB_READ_AR TO ROLE TABLEAU_SVC;

-- Grant warehouse access
GRANT USAGE ON WAREHOUSE BI_WH TO ROLE TABLEAU_SVC;

-- Roll up to SYSADMIN
GRANT ROLE TABLEAU_SVC TO ROLE SYSADMIN;

-- Create service user (TYPE = SERVICE prevents password login)
USE ROLE USERADMIN;
CREATE USER TABLEAU_USER
    TYPE = SERVICE
    DEFAULT_ROLE = TABLEAU_SVC
    DEFAULT_WAREHOUSE = BI_WH
    COMMENT = 'Service account for Tableau';

USE ROLE SECURITYADMIN;
GRANT ROLE TABLEAU_SVC TO USER TABLEAU_USER;
Service User Type

Create service accounts with TYPE = SERVICE. This prevents password-based login and MFA requirements—services authenticate via key pairs or OAuth. Never grant service roles to human users.

Anti-Patterns to Avoid

Don't Do This
  • Granting access roles directly to users — Always use functional roles as intermediaries
  • Creating functional-to-functional role grants — Keep each functional role independent
  • Mixing human and service accounts — Use separate roles for each
  • Skipping the access role layer — It provides reusability and reduces duplication

Why Use the Three-Tier Model?

Scenario Without Three-Tier With Three-Tier
New analyst joins Grant 20+ individual privileges Grant 1 functional role
Database permissions change Update grants for every user Update 1 access role
Audit who has access Check every user's grants Check role memberships
New BI tool added Copy grants from another service Compose from existing access roles
