Database Roles

When to use database-scoped roles and how they enable granular data sharing.

Database Roles vs. Account Roles

Database roles are scoped to a single database, while account roles are global. Here's when to use each:

Aspect Account Roles Database Roles
Scope Entire account Single database only
Can grant privileges on Any object in account Objects within its database only
Can be granted to Users, other account roles Account roles only (NOT users directly)
Can be activated in session Yes (USE ROLE) No (must go through account role)
Ideal for Cross-database access, user assignments Database encapsulation, data sharing
Key Limitation

Database roles cannot be granted directly to users. They must be granted to an account role first, and then that account role is granted to users.

When to Use Database Roles

Data Sharing

Share specific objects with data consumers while hiding others in the same database. Essential for Snowflake Marketplace listings.

Database Encapsulation

Let database owners manage their own access without needing account-level SECURITYADMIN privileges.

Multi-Tenant Isolation

Separate tenant access within a shared database structure using database roles per tenant.

Schema-Level Control

Different access levels for different schemas within one database, without creating account-level complexity.

Creating Database Roles

Database roles are created inside a database using a fully qualified name:

-- Create database roles (must specify database)
CREATE DATABASE ROLE ANALYTICS_DB.REPORTING_READ;
CREATE DATABASE ROLE ANALYTICS_DB.REPORTING_WRITE;
CREATE DATABASE ROLE ANALYTICS_DB.FINANCE_SCHEMA_READ;

-- Grant privileges to database roles
GRANT USAGE ON SCHEMA ANALYTICS_DB.REPORTING 
    TO DATABASE ROLE ANALYTICS_DB.REPORTING_READ;
GRANT SELECT ON ALL TABLES IN SCHEMA ANALYTICS_DB.REPORTING 
    TO DATABASE ROLE ANALYTICS_DB.REPORTING_READ;
GRANT SELECT ON FUTURE TABLES IN SCHEMA ANALYTICS_DB.REPORTING 
    TO DATABASE ROLE ANALYTICS_DB.REPORTING_READ;

-- Grant database role to an account role (REQUIRED step)
GRANT DATABASE ROLE ANALYTICS_DB.REPORTING_READ 
    TO ROLE DATA_ANALYST_FR;
Implicit USAGE Grant

When you grant a database role to an account role, Snowflake automatically grants USAGE on the database. You don't need to grant it separately.

Database Roles for Data Sharing

Database roles enable granular data sharing. Instead of sharing entire databases, you can share specific database roles:

-- =====================================================
-- Provider Account: Set up granular sharing
-- =====================================================

-- Create database roles for different partners
CREATE DATABASE ROLE SHARED_DB.PARTNER_A_ACCESS;
CREATE DATABASE ROLE SHARED_DB.PARTNER_B_ACCESS;
CREATE DATABASE ROLE SHARED_DB.PARTNER_C_ACCESS;

-- Grant different objects to different partners
GRANT SELECT ON TABLE SHARED_DB.PUBLIC.PRODUCT_CATALOG 
    TO DATABASE ROLE SHARED_DB.PARTNER_A_ACCESS;
GRANT SELECT ON TABLE SHARED_DB.PUBLIC.PRICING_PUBLIC 
    TO DATABASE ROLE SHARED_DB.PARTNER_A_ACCESS;

GRANT SELECT ON TABLE SHARED_DB.PUBLIC.INVENTORY 
    TO DATABASE ROLE SHARED_DB.PARTNER_B_ACCESS;

GRANT SELECT ON ALL TABLES IN SCHEMA SHARED_DB.ANALYTICS 
    TO DATABASE ROLE SHARED_DB.PARTNER_C_ACCESS;

-- Create share and add database roles
CREATE SHARE PARTNER_DATA_SHARE;
GRANT USAGE ON DATABASE SHARED_DB TO SHARE PARTNER_DATA_SHARE;
GRANT DATABASE ROLE SHARED_DB.PARTNER_A_ACCESS TO SHARE PARTNER_DATA_SHARE;
GRANT DATABASE ROLE SHARED_DB.PARTNER_B_ACCESS TO SHARE PARTNER_DATA_SHARE;
GRANT DATABASE ROLE SHARED_DB.PARTNER_C_ACCESS TO SHARE PARTNER_DATA_SHARE;

-- =====================================================
-- Consumer Account: Access shared data
-- =====================================================

-- Create database from share
CREATE DATABASE PARTNER_DATA FROM SHARE provider_account.PARTNER_DATA_SHARE;

-- Grant specific database role to your users
GRANT DATABASE ROLE PARTNER_DATA.PARTNER_A_ACCESS 
    TO ROLE MY_ANALYST_ROLE;

Database Role Limitations

Important Constraints
  • Cannot be granted directly to users — Must go through an account role
  • Cannot be activated in a session — No USE DATABASE ROLE command
  • Account roles cannot be granted to database roles — Only the reverse
  • If granted to a share, cannot have other database roles granted to it

What You Can't Do

-- ❌ WRONG: Cannot grant database role directly to user
GRANT DATABASE ROLE ANALYTICS_DB.REPORTING_READ TO USER jane;
-- Error: Cannot grant a database role to a user

-- ❌ WRONG: Cannot use database role directly
USE DATABASE ROLE ANALYTICS_DB.REPORTING_READ;
-- Error: Invalid syntax

-- ❌ WRONG: Cannot grant account role to database role
GRANT ROLE MY_ACCOUNT_ROLE TO DATABASE ROLE ANALYTICS_DB.REPORTING_READ;
-- Error: Cannot grant an account role to a database role

The Correct Pattern

-- ✓ CORRECT: Grant database role to account role, then to user
GRANT DATABASE ROLE ANALYTICS_DB.REPORTING_READ 
    TO ROLE DATA_ANALYST_FR;

GRANT ROLE DATA_ANALYST_FR TO USER jane;

-- User activates the account role, which includes database role privileges
USE ROLE DATA_ANALYST_FR;

When to Use Account Roles Instead

For most enterprise deployments, account roles (as access roles) are simpler. Use database roles only when you need:

  • Granular data sharing with external accounts
  • Database owners to manage their own access
  • Isolation between databases in multi-tenant scenarios
Recommendation

Start with account roles for your access role layer. Only introduce database roles when you have a specific need—typically data sharing or database-level autonomy.
