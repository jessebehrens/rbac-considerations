Snowflake RBAC Framework

A framework for thinking through Role-Based Access Control decisions in Snowflake.

Core Principles

1. Reserve High-Privilege Roles

ACCOUNTADMIN and SECURITYADMIN should be used sparingly. Create custom admin roles (PLATFORM_ADMIN, FINOPS_ADMIN) for daily operations.

2. Separate Access from Function

Create Access Roles for object permissions. Create Functional Roles for job titles. Grant access roles to functional roles.

3. Plan Your Hierarchy Intentionally

Decide early whether custom roles should roll up to SYSADMIN. There's no one-size-fits-all answer—base this on your governance model, compliance needs, and operational structure.

4. Use Future Grants

Set up future grants so new objects automatically inherit correct permissions. Reduces ongoing maintenance.

Custom Admin Roles

These roles delegate specific administrative functions without granting full system role access.

PLATFORM_ADMIN

Manages warehouses, resource monitors, network policies, network rules, storage integrations, compute pools, session policies, password policies, authentication policies, notification integrations, tasks, and other Snowflake objects.

FINOPS_ADMIN

Monitors costs, manages budgets, and oversees resource consumption. Develops alerts for budget thresholds and cost anomalies. Has visibility into billing and usage metrics to optimize spend.

DATA_SHARING_ADMIN

Manages data shares, listings, and external data exchange. Controls where data is shared, but the database owner must separately grant objects to the share. This means DATA_SHARING_ADMIN can manage distribution without having query access to the data itself.
