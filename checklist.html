Implementation Checklist

Step-by-step checklist for initial setup, validation queries, and ongoing maintenance.

Pre-Implementation Checklist

Planning Phase 1

Document your data domains

List all databases/schemas and their owners (Sales, Marketing, Finance, etc.)

Define access levels per domain

Determine who needs read vs. write access to each domain

Map job functions to data needs

Create a matrix of job titles → data domains → access levels

Identify service accounts needed

List all tools/services that will connect: dbt, Tableau, Fivetran, etc.

Choose naming conventions

Decide on suffixes: _AR (access), _FR (functional), _SVC (service)

Plan IdP integration (if applicable)

Map IdP groups to Snowflake functional roles

Initial Setup Checklist

Access Roles Phase 2

Create READ access roles for each database

CREATE ROLE {DB_NAME}_READ_AR;

Create WRITE access roles for each database

CREATE ROLE {DB_NAME}_WRITE_AR;

Grant USAGE on databases and schemas

Remember: access requires USAGE at every level of the hierarchy

Grant object-level privileges (SELECT, INSERT, etc.)

Use GRANT ... ON ALL TABLES IN DATABASE

Set up FUTURE GRANTS for automatic permissions

Essential for new objects to inherit permissions

Functional & Service Roles Phase 2

Create functional roles for each job function

CREATE ROLE {TEAM}_{FUNCTION}_FR;

Grant access roles to functional roles

Compose functional roles from multiple access roles

Create service roles for each tool/service

CREATE ROLE {TOOL}_SVC;

Create service users with TYPE = SERVICE

Prevents password login, requires key pair auth

Roll up all custom roles to SYSADMIN

GRANT ROLE {role} TO ROLE SYSADMIN;

Platform Roles Phase 2

Create PLATFORM_ADMIN role

For warehouse/integration management without data access

Create FINOPS_ADMIN role

Grant access to SNOWFLAKE database for cost monitoring

Create SECURITY_ANALYST_FR role

Grant to SECURITYADMIN for audit capabilities

SCIM Integration (if applicable) Phase 2

Create SCIM security integration

For Okta, Azure AD, or custom SCIM provider

Grant SCIM provisioner role to SECURITYADMIN

Not ACCOUNTADMIN—keep ACCOUNTADMIN minimal

Map IdP groups to Snowflake functional roles

Create roles in Snowflake first, then map in IdP

Test with pilot users before full rollout

Verify provisioning and deprovisioning works correctly

Warehouses Phase 2

Create purpose-specific warehouses

DEV_WH, ANALYTICS_WH, ETL_WH, BI_WH

Create warehouse access roles

CREATE ROLE {WH_NAME}_USAGE_AR;

Grant warehouse access to functional/service roles

Don't forget: users need warehouse access to run queries!

Set up resource monitors for cost control

Create monitors and assign to warehouses

Validation Queries

Run these queries to verify your setup is correct:

Check Role Hierarchy

-- View all roles granted to SYSADMIN
SHOW GRANTS TO ROLE SYSADMIN;

-- Check if any custom roles are NOT granted to SYSADMIN (potential issue)
SELECT NAME 
FROM SNOWFLAKE.ACCOUNT_USAGE.ROLES
WHERE NAME NOT IN (
    'ACCOUNTADMIN', 'SECURITYADMIN', 'SYSADMIN', 
    'USERADMIN', 'PUBLIC', 'ORGADMIN'
)
AND NAME NOT IN (
    SELECT GRANTEE_NAME 
    FROM SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_ROLES 
    WHERE GRANTED_TO = 'ROLE' 
        AND GRANTED_ON = 'ROLE'
        AND NAME = 'SYSADMIN'
);

Check Future Grants

-- List all future grants in a database
SHOW FUTURE GRANTS IN DATABASE {your_database};

-- Verify future grants are set up for access roles
SELECT * FROM TABLE(RESULT_SCAN(LAST_QUERY_ID()))
WHERE "grant_to" = 'ROLE';

Verify User Access

-- Check what roles a user has
SHOW GRANTS TO USER {username};

-- Check effective privileges for a role
SHOW GRANTS TO ROLE {role_name};

-- Check who has access to a specific table
SHOW GRANTS ON TABLE {database}.{schema}.{table};

ACCOUNTADMIN Usage Audit

-- Check who has ACCOUNTADMIN
SELECT GRANTEE_NAME, CREATED_ON
FROM SNOWFLAKE.ACCOUNT_USAGE.GRANTS_TO_USERS
WHERE ROLE = 'ACCOUNTADMIN'
    AND DELETED_ON IS NULL;

-- Check recent ACCOUNTADMIN usage
SELECT 
    USER_NAME,
    QUERY_TEXT,
    START_TIME
FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY
WHERE ROLE_NAME = 'ACCOUNTADMIN'
    AND START_TIME >= DATEADD(DAY, -30, CURRENT_TIMESTAMP())
ORDER BY START_TIME DESC
LIMIT 100;

Ongoing Maintenance Checklist

Weekly/Monthly Tasks Ongoing

Review ACCOUNTADMIN usage

Should be minimal—investigate unexpected usage

Check for orphaned roles

Roles not granted to any user or not rolling up to SYSADMIN

Review new user provisioning

Ensure new users are getting correct roles via SCIM

Audit sensitive data access

Review who accessed PII or financial data

Quarterly Tasks Ongoing

Review role membership

Remove users who no longer need access

Validate role hierarchy diagram

Ensure documentation matches actual implementation

Review service account credentials

Rotate keys, review access patterns

Test disaster recovery procedures

Ensure you can recreate roles if needed

Common Issues & Solutions

"User cannot access table"

Solution: Check USAGE grants on database and schema, not just SELECT on table. Run SHOW GRANTS TO ROLE {role} to see what's missing.

"New tables aren't accessible"

Solution: Verify future grants are set up. Run SHOW FUTURE GRANTS IN DATABASE {db}. Remember: schema-level future grants override database-level.

"Cannot grant privileges on object"

Solution: Check object ownership. Only the owner or SECURITYADMIN (with MANAGE GRANTS) can grant. For managed access schemas, only schema owner or SECURITYADMIN can grant.

"SCIM not syncing users"

Solution: Check SCIM integration status with SHOW SECURITY INTEGRATIONS. Verify token hasn't expired. Check IdP configuration matches Snowflake integration settings.
